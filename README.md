# Broadband Fraud Batch API

This service scores stored broadband transactions in MongoDB and returns a fraud label for each record plus a summary. It is batch-only: there is no live single-record prediction endpoint.

## How it works

```
MongoDB collection → batch fetch → validation → rule engine → XGBoost model
       → ensemble decision → per-record fraud flag + summary
```

## Project structure

```
fraud_api/
├── app/
│   ├── main.py              # FastAPI app + /predict batch route, /health
│   ├── models/
│   │   ├── schemas.py       # Pydantic models for MongoDB records and batch responses
│   │   └── fraud_model.json # trained XGBoost model (generated by training script)
│   ├── services/
│   │   ├── rules.py         # rule-based scoring
│   │   ├── features.py      # shared feature engineering (train + inference)
│   │   ├── ml.py             # XGBoost loading + inference
│   │   └── ensemble.py      # combine rule_score + ml_score
│   ├── config/
│   │   └── rules.yaml       # editable rule thresholds (no redeploy needed)
│   └── core/
│       └── logging.py
├── training/
│   └── train_model.py       # offline training script (synthetic data included)
├── tests/
└── requirements.txt
```

## Setup

```bash
python -m venv venv
source venv/bin/activate       # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

## 1. Configure MongoDB

Create a local `.env` file in the project root:

```bash
MONGODB_URI=mongodb://localhost:27017
MONGODB_DB=fraud_api
MONGODB_COLLECTION=transactions
```

If you use MongoDB Atlas, replace `MONGODB_URI` with your connection string.

## 2. Train the model (creates app/models/fraud_model.json)

```bash
python training/train_model.py
```

This uses **synthetic data** so you can run the whole project end-to-end
immediately. Replace `generate_synthetic_data()` in `training/train_model.py`
with a loader for your real, labeled historical data before using this for
anything real.

## 3. Run the API

```bash
uvicorn app.main:app --reload
```

Visit `http://127.0.0.1:8000/docs` for interactive Swagger docs.

## 4. Add MongoDB documents

Store transaction records in the configured collection using this shape:

```json
{
  "customer_id": "CUST-10293",
  "usage_mb": 15230.5,
  "avg_usage_mb": 2100.0,
  "device_age_days": 0,
  "num_devices_30d": 4,
  "failed_payments_7d": 3,
  "account_age_days": 5,
  "login_hour": 3,
  "distance_from_usual_km": 120.0,
  "mac_address": "AA:BB:CC:DD:EE:FF"
}
```

## 5. Run a batch prediction

```bash
curl -X POST http://127.0.0.1:8000/predict \
  -H "Content-Type: application/json" \
  -d '{
    "collection_name": "transactions",
    "customer_id": null,
    "skip": 0,
    "limit": 100
  }'
```

The endpoint fetches matching documents from MongoDB, scores them one by one, and returns the predictions plus a summary.

Example batch response:

```json
{
  "collection_name": "transactions",
  "matched_count": 2,
  "predictions": [
    {
      "document_id": "66b3f1b1c2d4e5f678901234",
      "customer_id": "CUST-10293",
      "is_fraud": true,
      "rule_score": 0.9,
      "ml_score": 0.87,
      "final_score": 1.0,
      "decision": "BLOCK",
      "triggered_rules": ["blacklisted_device", "usage_spike"],
      "model_version": "v1.0.0"
    }
  ],
  "summary": {
    "total_records": 2,
    "fraud_count": 1,
    "not_fraud_count": 1,
    "average_rule_score": 0.45,
    "average_ml_score": 0.5,
    "average_final_score": 0.75
  }
}
```

The `is_fraud` flag is the binary result for each record. The `decision` field is still included for traceability.

## Editing rule thresholds

Edit `app/config/rules.yaml` to change scoring thresholds. Restart the app after editing.

## Validation

Run the test suite with:

```bash
python -m pytest
```
